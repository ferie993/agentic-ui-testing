# Agentic UI Testing Framework 🤖✨

Welcome to the **Agentic UI Testing** repository. This project demonstrates a cutting-edge approach to automated web application testing by combining **Natural Language (Gherkin)**, **Gemini CLI**, and **Playwright CLI**. 

Instead of manually writing brittle CSS and XPath selectors, you simply describe the test scenario in natural language. An autonomous AI agent then acts as the automation engineer: it reads your scenario, understands the web page context, generates the exact Playwright script required, executes it, and validates the results.

---

## 🌟 Key Features

- **Agentic Generation:** AI autonomously writes and executes Playwright automation scripts based on raw Gherkin `.feature` files.
- **Smart Caching:** Generated step definitions are cached to disk. Subsequent runs execute instantly (< 3 seconds) without AI latency or token costs.
- **Self-Healing Fallback:** If the web app changes and the cache becomes stale or fails, the AI automatically kicks in, reads the new DOM, regenerates the script, and retries.
- **Dynamic Tagging:** Run selective suites using XRAY tags or Issue Keys (e.g., `@smoke, @regression`).
- **Comprehensive Reporting:** Integrated with Allure Report to provide beautiful, clear execution histories and screenshots of failures.
- **Advanced Assertions:** AI is capable of handling exact string matching, DOM state visibility checks, and even complex logical checks like Date Sorting (Ascending/Descending).

## 📂 Project Structure

This project is meticulously organized to separate test definitions, AI logic, execution engines, and reporting.

```text
agentic-ui-testing/
├── .env                  # Environment variables (API keys, Base URL, Tags).
├── Makefile              # Centralized commands for installing and running tests.
├── README.md             # This documentation file.
├── config/               # Configuration files for reporting templates.
├── helper/               # The core logic of the Agentic workflow.
│   ├── agent-logic.sh    # The "Brain": Contains the Gemini AI prompt engineering and interaction rules.
│   └── runner.sh         # The "Muscle": Handles file discovery, cache validation, execution, and Allure generation.
├── step-definitions/     # The "Cache": Automatically populated with Bash/Playwright scripts generated by the AI.
├── test/                 # The "Requirements": Your Gherkin `.feature` files describing user scenarios.
└── allure-results/       # Raw JSON and screenshot data generated during test execution.
```

### Deep Dive: The Core Components
1. **`test/*.feature`**: Where you write what needs to be tested in Indonesian/English (e.g., `Given pengguna berada pada halaman...`).
2. **`helper/agent-logic.sh`**: If a test hasn't been run before (Cache Miss), this script calls `gemini` with a strict `PRECISION PROTOCOL`. It teaches the AI how to use `npx playwright-cli` safely, how to handle dropdowns, and how to assert states.
3. **`helper/runner.sh`**: The orchestrator. It parses multiple comma-separated tags, finds matching `.feature` files, checks if a cached `.sh` script exists in `step-definitions/`, and executes it. It also generates the JSON files required by Allure.

## 🚀 Getting Started

### 1. Prerequisites
- **Node.js** (v18+) & **npm**  
  *How to install:*
  - **Windows/macOS/Linux:** It is highly recommended to use a version manager like [nvm](https://github.com/nvm-sh/nvm) (macOS/Linux) or [nvm-windows](https://github.com/coreybutler/nvm-windows) (Windows).
  - **Direct Download:** Alternatively, download the LTS version from the [official Node.js website](https://nodejs.org/).
- **Gemini CLI** (`npm install -g @google/gemini-cli`)
- **Playwright CLI** (`npm install -g @google/playwright-cli`)
- **Allure Commandline** (for viewing reports: `npm install -g allure-commandline`)

### 2. Installation
Clone the repository to your local machine:

```bash
git clone https://gitlab.com/wartek-id/matk/agentic-ui-testing.git
cd agentic-ui-testing
```

### 3. Environment Configuration
Copy the template configuration file to create your own `.env` file:

```bash
cp .env.template .env
```

Open `.env` and configure your variables:
```env
GEMINI_API_KEY="your-gemini-api-key"

# The target URL of the application you want to test
BASE_URL="https://rumah.pendidikan.go.id/"

# The specific tests you want to run (comma separated)
XRAY_ISSUE_KEY="RP-345,RP-355"
XRAY_TAGS="@smoke"
```

## 🧠 AI Model Configuration

In this `agentic-ui-testing` project, the AI model orchestrating the test generation inside `helper/agent-logic.sh` is:

**`gemini-3-flash-preview`**

**Technical Configuration Details:**
1. **Default Model Routing (`auto`)**: The `agent-logic.sh` script invokes the `gemini` command without explicitly specifying a model flag (`-m`). By default, the latest Gemini CLI utilizes the `auto` alias.
2. **Model Resolution**: For complex prompt engineering and coding logic tasks—like the precise DOM manipulation and validation generated in this project—the `auto` alias currently routes to **Gemini 3 Flash Preview**. This offers the perfect balance between blazing-fast execution speeds and high-accuracy reasoning.
3. **Project Overrides**: The `.env` file intentionally omits a `GEMINI_MODEL` variable, allowing the project to rely on the CLI's native, optimized auto-routing. If you ever wish to force a different model (e.g., `gemini-3-pro-preview`), simply add `GEMINI_MODEL="gemini-3-pro-preview"` to your `.env` file.

## ⚙️ Running the Automation

The execution is entirely driven by the `Makefile`, making it incredibly simple to run suites based on the variables defined in your `.env` file. 

The framework supports multiple tags separated by commas (e.g., `XRAY_TAGS="@smoke,@regression"`).

### Run by Tags
Executes all `.feature` files inside `test/` that contain the tags specified in `XRAY_TAGS`.

- **Headless Mode** (Fastest, hidden browser - ideal for CI/CD):
  ```bash
  make test-tags-headless
  ```
- **Headed Mode** (Visible browser - ideal for debugging):
  ```bash
  make test-tags-headed
  ```

### Run by Issue Keys
Executes specific test scenarios based on tracking IDs specified in `XRAY_ISSUE_KEY`.

- **Headless Mode** (Fastest, hidden browser - ideal for CI/CD):
  ```bash
  make test-issue-headless
  ```
- **Headed Mode** (Visible browser - ideal for debugging):
  ```bash
  make test-issue-headed
  ```

## 🏗️ System Architecture

The relationship between the `Makefile`, `helper/runner.sh`, and `helper/agent-logic.sh` in this project forms a smart automation pipeline (Agentic Workflow).

They work together like a team, where each component has a specific role: the `Makefile` acts as the user interface (the launch button), `runner.sh` as the field supervisor (executor & reporter), and `agent-logic.sh` as the AI brain (script generator).

Here is an in-depth explanation of their logic and workflow:

### 1. `Makefile` (The Interface / Entry Point)
The `Makefile` is the starting point of your interaction with this framework. Instead of typing long and complex Bash commands every time you want to run tests, the `Makefile` simplifies them into short commands (like `make test-tags-headless`).

**Main Tasks:**
- **Read Configuration:** Retrieves all variables you set in the `.env` file (such as `XRAY_TAGS`, `XRAY_ISSUE_KEY`, and `BASE_URL`).
- **Sanitize Input:** Converts comma-separated tags or issue keys input (e.g., `@smoke,@regression`) into a space-separated format so that the underlying scripts can easily read them.
- **Trigger Process:** Calls and executes the `helper/agent-logic.sh` script, passing the sanitized parameters along with the selected browser mode (HEADED or HEADLESS).

---

### 2. `helper/agent-logic.sh` (The AI Brain / Prompt Engineer)
This is the main "Brain" of the Agentic UI Testing project. The `Makefile` calls this file first, not the runner. This file is fully responsible for ensuring every test scenario has a ready-to-execute Playwright script.

**Main Tasks:**
- **Initial Delegation:** When called by the `Makefile`, the very first thing `agent-logic.sh` does is ask `helper/runner.sh` to attempt running the tests.
- **Failure Detection (Cache Miss):** If `runner.sh` reports back that there are Gherkin scenarios (`.feature`) that do not yet have a `.sh` script (Cache Miss) or the script is outdated (Cache Stale), then `agent-logic.sh` takes over control.
- **Prompt Engineering & Calling Gemini:** This is where the magic happens. The script wraps the names of the missing `.feature` files into a very strict and comprehensive instruction (Prompt Protocol). This prompt contains rules on how to click, how to verify text (Exact match, Visibility, Ascending/Descending), and how to take screenshots.
- **Code Generation:** The prompt is sent to the Gemini CLI. Gemini reads the Gherkin files, understands the natural language, and directly writes the Bash/Playwright code into the `step-definitions/` folder.
- **Re-Execution:** Once the AI finishes generating the code, `agent-logic.sh` calls `runner.sh` once more to execute the newly created scripts.

---

### 3. `helper/runner.sh` (The Supervisor / Executor & Reporter)
This script is the workhorse that handles all non-AI logic. Its job is purely operational: finding files, executing them, and recording the results.

**Main Tasks:**
- **URL Health Check:** Uses `curl` to ensure the `BASE_URL` (e.g., `rumah.pendidikan.go.id`) is accessible and not down before the tests begin.
- **Scenario Discovery:** Reads the tag parameters from `agent-logic.sh` and scans the `test/` directory to find which `.feature` files contain those tags.
- **Cache Validation:** Checks the `step-definitions/` folder. If a `.sh` script for that `.feature` file already exists, it marks it as Valid. If it doesn't exist (or the `.feature` file was recently modified), it marks it as Stale and reports back to `agent-logic.sh` (with Exit Code 2 or 4) to have it regenerated by the AI.
- **Lightning Execution:** For Valid scripts, it immediately executes them (`bash step-definitions/...-step.sh`) in milliseconds.
- **Report Generation (Allure):** During execution, `runner.sh` calculates the start/stop time, captures error messages, links screenshot images, and compiles everything into a `.json` file in the `allure-results/` folder so it can be visualized as a graphical report by Allure.

## 🔄 How the Agentic Loop Works (The Loop)

1. **Trigger**: You run `make test-tags-headless`.
2. **Discovery**: `runner.sh` scans the `test/` directory for `.feature` files matching your tags.
3. **Cache Validation**: It checks if a corresponding script exists in `step-definitions/`.
   - **Cache HIT**: The test runs immediately using the cached script.
   - **Cache MISS**: The execution halts and passes control to the AI.
4. **AI Generation**: `agent-logic.sh` prompts Gemini CLI with the missing `.feature` files. Gemini analyzes the English/Indonesian steps and writes a flawless Playwright script.
5. **Execution**: The runner automatically restarts and executes the newly generated, cached script.
6. **Reporting**: Results, including screenshots of failures, are compiled into `allure-results/`.
